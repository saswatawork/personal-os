# ADR-009: GitHub Profile README Auto-Updater

## Date
2026-06-23

## Status
Accepted

## Context
Saswata's work — video studio, personal-os, LLM pipeline architecture — is strong enough
to attract recruiter and hiring manager attention. But none of it is visible. The GitHub
profile is the one public surface that engineering hiring teams actually look at.

The problem with manually writing a profile README: it goes stale, it never gets updated,
and writing about yourself is uncomfortable so it doesn't happen. Automating it removes
the friction entirely.

## Decision

Build `apps/job_search/github_readme.py` — a script that:
1. Loads personal context (profile, career, goals) via `core/context_loader.py`
2. Calls the LLM via `core/provider.py` with a specific prompt that generates a
   GitHub profile README in simple, plain English
3. Pushes the result to the `saswatapal/saswatapal` GitHub profile repo via PyGithub
4. Can be run manually or on a cron schedule

## Reasoning

**AI-generated over static template:**
The README should reflect current context — what he's focused on, what he's building toward.
A static template goes stale. The LLM reads the context files (which the user keeps current)
and generates something that accurately reflects where he is right now. Updating the profile
becomes: update a context file, rerun the script.

**Simple English over technical jargon:**
GitHub profiles are scanned by recruiters and engineering managers, not just engineers.
"I build LLM pipelines" lands worse than "I build systems where multiple AI models work
together to get a job done." The prompt explicitly instructs: no buzzwords, plain sentences,
explain the why behind the what.

**PyGithub over raw API calls:**
Official GitHub REST API is well-supported, no auth complexity, one library.
PyGithub is the standard Python wrapper — handles SHA management for file updates automatically.

**Fits the existing architecture:**
Uses ContextLoader and Provider exactly as designed. No new patterns introduced.
Personal context stays local. GitHub token comes from env var.

## Alternatives Considered

**Static markdown template with variables:**
Rejected. Requires manual updates to stay accurate. Defeats the purpose of automation.

**GitHub Actions to regenerate:**
Rejected for now. Adds GitHub-side complexity and requires secrets management in GitHub.
Running locally is simpler and gives full control. Can add GH Actions later if needed.

**Scraping pinned repos dynamically:**
Rejected for first version. Adds complexity (parsing repo metadata, commit counts, etc.)
without clear benefit. The LLM generates the content — it doesn't need live repo data
to write a compelling profile. Add repo stats in v2 if useful.

## Tradeoffs

**Gain:**
- Profile stays current with zero manual effort
- Content is generated by reasoning about actual goals, not just listing skills
- Simple English makes it accessible to non-technical readers (recruiters)
- Completely automated — run once, cron the rest

**Give up:**
- LLM output is non-deterministic — each run produces slightly different text
- Requires a GitHub personal access token with repo write scope
- Requires PyGithub installed

## Future Implications
- Same `apps/job_search/` module will house LinkedIn and Naukri automation later
- LinkedIn posting script will use a similar pattern: context → prompt → publish
- Could add a `--dry-run` flag to preview without pushing — useful before enabling cron
